     ---
     Phase 1 — Wire Up Existing Internals (Low Effort, High Value)

     These are cases where the logic exists but isn't fully connected.

     1.1 OAuth Token Introspection Endpoint

     Problem: OAuthProvider has a -introspectToken:completion: method and the
      introspection_endpoint URL is advertised in the OAuth server metadata
     (OAuthProvider.m:561) — but POST /oauth/introspect is never registered as an
      HTTP route. Clients that call the advertised introspection endpoint get a 404.

     Fix: Add the route registration in OAuth2Handler.m inside
     -registerRoutesWithServer:. Wire it to the existing
     -introspectToken:completion: on OAuthProvider. Return the standard RFC 7662
     JSON body (active, sub, scope, client_id, exp, etc.).

     Files:
     - ATProtoPDS/Sources/Auth/OAuth2Handler.m — add POST /oauth/introspect route
      alongside the existing token/revoke routes (~line 1497)
     - ATProtoPDS/Sources/OAuthProvider/OAuthProvider.m — existing
     introspectToken:completion: (line 524) can be called directly

     Test: OAuthConformanceTests.m — add a test that calls the advertised
     introspection_endpoint and receives a valid RFC 7662 response.

     ---
     1.2 com.atproto.repo.getBlob — Add Range Support

     Problem: repo.getBlob (XrpcRepoMethods.m:584-647) has its own
     implementation that returns plain 200 OK with no Accept-Ranges or 206
     support. The spec endpoint sync.getBlob has full Range support. Having two
     blob endpoints with different capabilities is a bug.

     Fix: Extract the Range-parsing and partial-content response logic from
     sync.getBlob (XrpcSyncMethods.m:924-993) into a shared helper (e.g., a
     method on BlobStorage or a static helper in a BlobResponseBuilder). Then
     call it from both repo.getBlob and sync.getBlob.

     Files:
     - ATProtoPDS/Sources/Network/XrpcSyncMethods.m — extract
     parseByteRangeHeader() and the 206 response block
     - ATProtoPDS/Sources/Network/XrpcRepoMethods.m — call the shared helper (line
     - ATProtoPDS/Sources/Blob/BlobStorage.h/.m — best home for a
     -respondWithBlob:request:response: helper

     Test: BlobXrpcTests.m — add Range request tests for repo.getBlob
     mirroring existing sync.getBlob range tests.

     Phase 2 — Add Missing Spec Endpoints (New Functionality)

     2.1 app.bsky.draft.* — Stub Registration

     Problem: The 2024 draft lexicons (createDraft, deleteDraft, getDrafts,
      updateDraft) are in the official AT Protocol lexicon set but garazyk has no
     registration at all. A client that queries com.atproto.server.describeServer
     will not see these proxied or acknowledged.

     Decision: Implement as proxyOrNotSupported: (same pattern as
     app.bsky.ageassurance.* and app.bsky.contact.*) — proxy to an upstream
     AppView if configured, otherwise return 501.

     Files:
     - ATProtoPDS/Sources/Network/XRPC/XrpcAppBskyMethods.m — add 4 registrations
     using proxyOrNotSupported: alongside the existing ageassurance and contact
      blocks

     No new handler files needed — the proxyOrNotSupported: pattern already
     handles both cases inline.

     Phase 3 — Cleanup: Remove Non-Standard Methods

     These are methods registered under com.atproto.* namespaces that do not exist
     in the AT Protocol lexicons. No spec-compliant client will ever call them; they
     only add noise and create maintenance surface.

     3.1 com.atproto.server.getAccount

     What it does: Returns the authenticated user's account dictionary.
     Functionally duplicates com.atproto.server.getSession.

     Action: Remove the route registration and handler from XrpcServerMethods.m
      (~line 1205). The admin-facing equivalent (com.atproto.admin.getAccountInfo)
     already covers this for admin use.

     Files: ATProtoPDS/Sources/Network/XRPC/XrpcServerMethods.m

     ---
     3.2 com.atproto.repo.updateRecord

     What it does: A "legacy compatibility wrapper" around applyWrites that
     verifies the record exists before calling update. The spec endpoint for record
     updates is putRecord.

     Action: Remove the route registration and handler from XrpcRepoMethods.m
     (~line 923). Any code paths internally that use this should call putRecord or
     applyWrites directly.

     Files: ATProtoPDS/Sources/Network/XRPC/XrpcRepoMethods.m

     ---
     3.3 com.atproto.repo.deleteBlob

     What it does: Deletes a blob by CID for the authenticated user. Not in the
     AT Protocol spec — blob lifecycle is managed through record references, not
     explicit deletion.

     Action: Remove from XrpcRepoMethods.m (~line 334). If blob GC is needed it
      should be an internal admin operation, not a public XRPC endpoint.

     Files: ATProtoPDS/Sources/Network/XRPC/XrpcRepoMethods.m

     ---
     3.4 com.atproto.repo.getBlob → Redirect or Remove

     What it does: Auth-gated blob retrieval, duplicating sync.getBlob with
     added auth enforcement.

     Options:
     - A) Keep it but make it delegate to the sync.getBlob implementation (after
     Phase 1.2 adds Range support) — preserves backward compat for any internal
     tooling
     - B) Remove it and point any callers to sync.getBlob

     Recommendation: Option A — keep as a thin auth-gated wrapper that calls the
     shared blob response helper from Phase 1.2, so Range support is consistent. The
     auth gate is the only meaningful difference and may be useful for
     private/unlisted blobs.

     Files: ATProtoPDS/Sources/Network/XRPC/XrpcRepoMethods.m

     ---
     3.5 com.atproto.label.createLabel + getLabels

     What they do: Admin-only endpoints for creating and querying labels. These
     are internal operations not exposed in the public AT Protocol spec.

     Action: Move these to the /admin/* HTTP path or prefix them
     /_admin/label/* so they are clearly internal. Alternatively, keep under
     com.atproto.label.* but add a note that they are non-standard internal
     extensions. Do not return 501 — they are still useful for self-hosted admin
     workflows. The key fix is making sure they are clearly not part of the public
     protocol surface.

     Files: ATProtoPDS/Sources/Network/XRPC/XrpcLabelMethods.m

     ---
     Phase 5 — S3 Blob Provider

     Problem: PDSBlobProvider.h protocol explicitly anticipates S3
     ("Implementations can store blobs on disk, S3, or other storage systems") and
     three of the four comparison implementations (cocoon, tranquil-pds, pegasus)
     have S3 support. Disk-only storage is a scaling bottleneck.

     Implementation approach:
     1. Create PDSCloudStorageBlobProvider.h/.m conforming to PDSBlobProvider
     protocol
     2. Use NSURLSession for S3 API calls (AWS Signature V4, or S3-compatible via
     configurable endpoint URL for MinIO, Cloudflare R2, Backblaze B2)
     3. Configure via PDSConfiguration: blobStorage = "s3", s3Bucket,
     s3Region, s3Endpoint, s3KeyPrefix
     4. PDSBlobProviderFactory (new) reads config and returns the appropriate
     provider
     5. Add CDN redirect support: if cdnURL is configured, sync.getBlob returns
     302 to {cdnURL}/{cid} instead of proxying the bytes

     Files:
     - ATProtoPDS/Sources/Blob/PDSCloudStorageBlobProvider.h/.m — new
     - ATProtoPDS/Sources/Blob/PDSBlobProviderFactory.h/.m — new
     - ATProtoPDS/Sources/App/PDSConfiguration.h/.m — add S3 config fields
     - ATProtoPDS/Sources/Network/XRPC/XrpcSyncMethods.m — add CDN 302 redirect
     path in getBlob
     - ATProtoPDS/Sources/Network/XRPC/XrpcRepoMethods.m — same in repo.getBlob

     Test: New CloudStorageBlobProviderTests.m using a mock HTTP server or
     LocalStack.
