### CRITICAL

#### 1. No Dynamic Client Metadata Fetching
**Spec**: "The Authorization Server will receive the PAR request and use the `client_id` URL to resolve the client metadata document."
**Code**: `validateClient:` checks the database first, then falls back to inline `client_metadata` from the PAR body. It **never fetches** from the `client_id` URL.
**Impact**: Arbitrary third-party clients cannot authenticate. Only pre-registered clients or those passing inline metadata work. The spec requires the server to fetch `GET <client_id>` and validate the response as the client metadata document.
**Location**: `OAuth2Handler.m:124-191`

#### 2. `plain` PKCE Method Allowed
**Spec**: "The `plain` method is not allowed."
**Code**: `verifyCodeVerifier:` at `OAuth2.m:1367` accepts `"plain"`.
**Impact**: Weakens PKCE security. Should reject `plain` and require `S256`.
**Location**: `OAuth2.m:1366-1379`

#### 3. DPoP Nonce TTL Exceeds Spec Maximum
**Spec**: "Servers must rotate nonces periodically, with a maximum lifetime of 5 minutes."
**Code**: `PDSNonceManager.m:45` â€” nonce expiration is 10 minutes (600 seconds).
**Impact**: Nonces live twice as long as permitted. Should be â‰¤300 seconds.
**Location**: `PDSNonceManager.m:45`

#### 4. DPoP Nonces Are One-Time Use (Should Be Reusable)
**Spec**: "Servers may use the same nonce across all client sessions and across multiple requests at any point in time. Servers should accept recently-stale (old) nonces to make rotation smoother for clients with multiple concurrent requests in-flight."
**Code**: `PDSNonceManager.m:66` â€” nonces are removed after a single validation (`removeObjectForKey:`).
**Impact**: Every request forces a nonce-error-retry cycle. Clients with concurrent requests will fail unnecessarily. Nonces should be valid until they expire, not consumed on first use.
**Location**: `PDSNonceManager.m:55-71`

#### 5. Confidential Client JWT Assertion Not Verified During Token Exchange
**Spec**: "Confidential clients authenticate themselves using JWT client assertions" for PAR and "all token requests (both the initial token request and any subsequent refresh requests)."
**Code**: PAR validates `client_secret` (a static secret, not a JWT assertion). Token endpoint at `OAuth2Handler.m:1505-1590` does not appear to verify `client_assertion` / `client_assertion_type` from confidential clients.
**Impact**: Confidential clients cannot properly authenticate. The spec requires `client_assertion_type: urn:ietf:params:oauth:client-assertion-type:jwt-bearer` with a signed JWT, not a `client_secret`.
**Location**: `OAuth2Handler.m:1763-1780` (PAR), `OAuth2Handler.m:1505-1590` (token)

### MODERATE

#### 6. Protected Resource Metadata Format Non-Standard
**Spec** (draft-ietf-oauth-resource-metadata): `authorization_servers` should be an array of URL strings: `["https://issuer.example.com"]`
**Code**: Returns `authorization_servers` as an array of objects with nested `authorization_server` and `resource_servers` keys.
**Impact**: Standard OAuth clients and libraries won't parse this correctly.
**Location**: `OAuth2Handler.m:966-977`

#### 7. Missing `atproto` Scope Enforcement
**Spec**: "The special `atproto` scope is required for all atproto OAuth sessions." and "Clients should reject token responses if they don't contain a `scope` field, or if the `scope` field does not contain `atproto`."
**Code**: Authorization requests and token responses don't enforce that `atproto` is included in the requested scope. The default fallback scope is `OAuth2ScopeIdentify` (`atproto:identify`), not `atproto`.
**Impact**: Sessions could be created without the `atproto` scope, which violates the spec.
**Location**: `OAuth2.m:1256` (fallback scope), `OAuth2.h:32-36` (scope constants)

#### 8. Scope Constants Don't Match Spec
**Spec**: Scopes are `atproto`, `transition:generic`, `transition:chat.bsky`, `transition:email`
**Code**: Defines `atproto:identify`, `atproto:signin`, `atproto:repo_write`, `atproto:repo_read`, `atproto:profile` â€” these are custom and not in the AT Protocol spec.
**Impact**: Interop failure with standard AT Protocol clients that request `atproto` scope.
**Location**: `OAuth2.m:32-36`

#### 9. No `code_challenge` Reuse Prevention
**Spec**: "Authorization Servers must prevent reuse of `code_challenge` values across sessions (at least within some reasonable time frame, such as a 24 hour period)."
**Code**: No mechanism to track previously used code challenges.
**Impact**: Potential replay attacks using captured PKCE challenges.

#### 10. PKCE Not Mandatory
**Spec**: "PKCE is mandatory for all Authorization Requests."
**Code**: PKCE verification at `OAuth2.m:1192` is conditional â€” `if (request.codeVerifier && codeData[@"code_challenge"])`. If no challenge was sent, PKCE is skipped entirely. PAR only requires `code_challenge` for public clients.
**Impact**: A client omitting PKCE will succeed, violating the spec. Both public and confidential clients must use PKCE.
**Location**: `OAuth2.m:1192`, `OAuth2Handler.m:1824-1832`

#### 11. Authorization Code Reuse Not Fully Prevented
**Spec**: "Authorization Servers should reject reuse of a `code` value, and revoke any outstanding sessions and tokens associated with the earlier use of the `code` value."
**Code**: The code is removed after first use (`removeAuthorizationCode:`), but there's no session revocation if a code is replayed. The in-memory dictionary may have race conditions on concurrent lookups.
**Impact**: If a code is used twice before the first `removeAuthorizationCode:` completes, both could succeed.
**Location**: `OAuth2.m:1158-1218`

#### 12. In-Memory Session Storage
**Code**: `activeSessions` is an `NSMutableDictionary` in memory. PAR requests are persisted to SQLite, but active sessions are not.
**Impact**: All active sessions are lost on server restart. Refresh tokens become invalid.

#### 13. `transition:email` Scope Missing from Metadata
**Spec**: Lists `transition:email` as a transitional scope.
**Code**: `scopes_supported` includes `atproto`, `transition:generic`, `transition:chat.bsky` but omits `transition:email`.
**Location**: `OAuthServerMetadata.m:49-50`

### LOW

#### 14. No `DPoP-Nonce` Header on Token Response
**Spec**: "Clients must reject responses missing a `DPoP-Nonce` header."
**Code**: Token endpoint returns `DPoP-Nonce` only after PAR. The token response handler doesn't consistently set it.
**Impact**: Spec-compliant clients may reject token responses.

#### 15. Missing `Cache-Control` on Metadata Endpoints
**Spec**: Metadata endpoints should use appropriate caching headers.
**Code**: `Cache-Control: no-store` is set on PAR response but not on metadata endpoints.

#### 16. `client_secret` Flow in PAR
**Code**: PAR handler validates `client_secret` from body params (`OAuth2Handler.m:1763-1780`).
**Spec**: AT Protocol explicitly says "There is no `client_secret` shared between servers and clients." Confidential clients use JWT assertions instead.
**Impact**: This code path should not exist for AT Protocol OAuth.

#### 17. JWT Signing Algorithm Mismatch
**Code**: `JWTMinter.signingAlgorithm` is set to `ES256K` (secp256k1) at `OAuth2.m:1004,1023`.
**Spec**: DPoP and token signing should use `ES256` (P-256). `ES256K` is a different curve.
**Impact**: Tokens signed with `ES256K` may not be verifiable by standard AT Protocol clients expecting `ES256`.
