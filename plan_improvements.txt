# Best Practice Improvements Plan

## 1. HttpServer: Fix Pipelining Recursion (Stack Overflow Risk)
- `HttpServer.m`: Refactor `tryProcessRequestFromState:data:connection:` to return a boolean indicating if it consumed data and processed a request, or refactor `handleReceivedData:onConnection:` to use a `while` loop over `unconsumedData`.
- Actually, better: Change `tryProcessRequestFromState` to just try processing once and return `BOOL` if unconsumed data remains. Wrap it in a `while` loop in `handleReceivedData:onConnection:`.

## 2. WebSocketCodec: Fix O(N^2) Buffer Shifting
- `WebSocketCodec.m`: Introduce `NSUInteger offset = 0;`
- Instead of modifying `self.readBuffer` inside the `while` loop, use `offset` to index into the bytes: `const uint8_t *bytes = self.readBuffer.bytes + offset;`
- Update all length checks to account for `offset` (`self.readBuffer.length - offset`).
- At the end of the `while` loop, if `offset > 0`, do a single `replaceBytesInRange:NSMakeRange(0, offset)`.

## 3. WebSocketCodec: Reassemble Fragments
- Add properties to `WebSocketCodec`: `NSMutableArray<NSData *> *fragments` and `uint8_t fragmentOpcode`.
- When reading a frame, extract `BOOL fin = (firstByte & WS_FLAG_FIN) != 0;`
- If it's a data frame (opcode 1 or 2) and `fin == 0`, save the opcode and append the payload to `fragments`. DO NOT emit an event.
- If it's a continuation frame (opcode 0):
  - Append payload to `fragments`.
  - If `fin == 1`, join all fragments, emit the event using `fragmentOpcode`, and clear fragments.
- Control frames (opcode >= 8) can be interleaved and are always `fin == 1`. Process them normally even during fragmentation.

## 4. HttpChunkedBodyParser: Accurate Byte Consumption
- Check `HttpChunkedBodyParser.h/.m`.
- Modify `appendData:error:` to return the exact number of bytes consumed (or out param), so `Http1Parser` knows where the next pipelined request begins.
- Update `Http1Parser.m` to set `consumedOffset = bodyStart + bytesConsumed` instead of `self.buffer.length`.
